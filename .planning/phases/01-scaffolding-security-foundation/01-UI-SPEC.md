# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Scaffolding + Security Foundation.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase 1 UI scope is minimal:** One walking skeleton home page — a heading and a
> subheading — rendered by `app/page.tsx`. No interactive components, no cards, no
> form inputs. The primary value of this spec is establishing system-level design
> decisions (font, color system, spacing scale, typography scale) that Phases 3 and 4
> will inherit without renegotiation.

## Spacing Scale

Standard 8-point scale. Used via Tailwind utility classes (`p-2 = 8px`, `p-4 = 16px`, etc.).

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | p-1 / gap-1 | Icon gaps, inline padding |
| sm | 8px | p-2 / gap-2 | Compact element spacing |
| md | 16px | p-4 / gap-4 | Default element spacing, card padding |
| lg | 24px | p-6 / gap-6 | Section padding |
| xl | 32px | p-8 / gap-8 | Layout gaps, page-level margin |
| 2xl | 48px | p-12 / gap-12 | Major section breaks |
| 3xl | 64px | p-16 / gap-16 | Page-level vertical spacing |

Exceptions: none for Phase 1.

**Phase 1 application:** The skeleton `app/page.tsx` uses `min-h-screen p-8` (xl = 32px)
as the page container padding. Per RESEARCH.md code example:
`<main className="min-h-screen p-8">`.

## Typography

Geist Sans is loaded via `next/font/google`. All sizes use Tailwind text utilities.

| Role | Size | Weight | Line Height | Tailwind Class |
|------|------|--------|-------------|----------------|
| Body | 16px | 400 regular | 1.5 | `text-base font-normal leading-relaxed` |
| Label | 14px | 400 regular | 1.5 | `text-sm font-normal leading-relaxed` |
| Heading | 24px | 700 bold | 1.2 | `text-2xl font-bold leading-tight` |
| Display | 32px | 700 bold | 1.2 | `text-3xl font-bold leading-tight` |

**Phase 1 application:** The skeleton page uses `text-2xl font-bold` for the H1
("PR Review Copilot") and default body text for the subtitle ("Paste a GitHub PR URL
to get started."). Per RESEARCH.md code example.

**Forward-looking (Phase 3+):** severity badges (HIGH/MEDIUM/LOW) will use Label
(14px, 400). Card headings will use Heading (24px, 700). No additional sizes will be
introduced without amending this spec.

## Color

Neutral dark-on-light system. Tailwind's default gray scale is used directly — no
custom palette. This matches Tailwind v4 defaults from `create-next-app`.

| Role | Value | Tailwind Token | Usage |
|------|-------|----------------|-------|
| Dominant (60%) | #ffffff | `bg-white` / `bg-background` | Page background, main surface |
| Secondary (30%) | #f4f4f5 | `bg-zinc-100` | Cards, sidebar, nav (Phase 3+) |
| Accent (10%) | #18181b | `bg-zinc-900` / `text-zinc-900` | Primary CTA button, active badge borders |
| Destructive | #dc2626 | `text-red-600` / `bg-red-50` | Error messages only |

**Accent reserved for:**
1. Primary CTA button background (Phase 3+: "Analyze PR" button)
2. Active severity badge borders for HIGH severity (Phase 4+)
3. Navigation active state indicator (if nav exists in Phase 3+)

Accent is NOT used for: secondary labels, card borders, icons, hover states on
non-primary elements.

**Phase 1 application:** The skeleton page uses only Dominant (white background)
and default text color (`text-gray-600` for subtitle). No accent in Phase 1 UI.

**Dark mode:** Out of scope for v1. No dark mode variants needed.

## Interaction Contract

Phase 1 has no user interactions. The page is static markup only.

| Interaction | Behavior | Phase |
|-------------|----------|-------|
| Page load | Renders H1 + subtitle; no loading state | Phase 1 |
| PR URL input | Not present | Phase 3 |
| Analyze PR submit | Not present | Phase 3 |
| Review card render | Not present | Phase 3–4 |

**Accessibility baseline (established now, enforced in Phase 3+):**
- All interactive elements must have visible focus rings (`focus-visible:ring-2`)
- Minimum touch target size: 44px height for buttons
- Color contrast: minimum 4.5:1 for body text, 3:1 for large text (WCAG AA)
- Error messages must use `role="alert"` for screen reader announcement
