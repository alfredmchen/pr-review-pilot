## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (initialize during Phase 3 plan execution) |
| Preset | none (`npx shadcn@latest init` with defaults: New York style, neutral base color, CSS variables ON) |
| Component library | radix (via shadcn) |
| Icon library | lucide-react (default shadcn dependency — installed transitively by `npx shadcn@latest add` for icon-bearing components) |
| Font | Geist Sans (body + headings), Geist Mono (code/diff content if needed) — already wired in `app/layout.tsx` via `next/font/google` |

**shadcn primitives to install in this phase (commands belong in the plan, not here):**
`card`, `skeleton`, `alert`, `button`, `input`, `badge`.

**Why these six:**
- `card` — every result surface (8 cards) wraps in `<Card>` for consistent border/padding
- `skeleton` — FETCH-03's 100ms loading state uses `<Skeleton>` blocks (animate-pulse)
- `alert` — `PrivateRepoCard`, `NotFoundCard`, `PRSizeWarningCard` use `<Alert>` for the status-distinguishing visual treatment (icon + variant color)
- `button` — Submit button in `PRUrlForm`
- `input` — URL input in `PRUrlForm`
- `badge` — placeholder cards display the `prType` label as a `<Badge>` so dispatch is visually verifiable in Phase 3 even before content lands

## Spacing Scale

Declared values (8-point scale, Tailwind v4 defaults — multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (Tailwind `gap-1`, `p-1`) | Icon-to-text gap inside `<Alert>`, badge inner padding |
| sm | 8px (Tailwind `gap-2`, `p-2`) | Compact spacing inside cards (label-to-value) |
| md | 16px (Tailwind `gap-4`, `p-4`) | Default card content padding, gap between form input and submit button |
| lg | 24px (Tailwind `gap-6`, `p-6`) | Card outer padding, gap between input form and result panel |
| xl | 32px (Tailwind `gap-8`, `p-8`) | Page main padding (matches existing `p-8` on `<main>`) |
| 2xl | 48px (Tailwind `gap-12`) | Vertical breathing room between H1 and form on desktop |
| 3xl | 64px (Tailwind `gap-16`) | Not used in Phase 3 |

**Layout-level spacing rules:**
- Main container: `max-w-3xl mx-auto p-8` — single-column, centered, ~768px max width
- H1 → subtitle: `mt-2` (8px)
- Subtitle → form: `mt-8` (32px)
- Form → result panel: `mt-6` (24px)
- Card → card (none in Phase 3 — only one result rendered at a time)
- Inside cards: `p-6` outer, `space-y-4` between sections (16px)

**Touch target exception:** Submit button minimum height is 40px (Tailwind `h-10`, shadcn default for `<Button>` size=`default`). Below the 44px ideal, but acceptable for desktop-primary demo per auto-mode constraints.

Exceptions: none beyond the touch-target note above.

## Typography

Phase 3 uses exactly **3 sizes** and **2 weights**. Heading and Display are the same role on this single-page demo.

| Role | Size | Weight | Line Height | Tailwind class |
|------|------|--------|-------------|----------------|
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-normal` |
| Label | 14px | 600 (semibold) | 1.4 | `text-sm font-semibold leading-snug` |
| Heading (card title) | 20px | 600 (semibold) | 1.3 | `text-xl font-semibold leading-snug` |
| Display (page H1) | 28px | 600 (semibold) | 1.2 | `text-3xl font-semibold tracking-tight` |

**Notes:**
- Existing `app/page.tsx` H1 uses `text-2xl font-bold` — Phase 3 upgrades to `text-3xl font-semibold` for the Display role above. Bold (700) is replaced by semibold (600) to honor the "exactly 2 weights" rule.
- Subtitle ("Paste a GitHub PR URL to get started.") is Body role.
- Inline error message inside `PRUrlForm` (invalid-url status) is Label role with `text-destructive` color.
- `prType` badge label and metadata labels ("Files changed", "Author") are Label role.
- Mono font (`font-mono`) is reserved for repo path / PR number when displayed; not invoked in Phase 3 placeholder cards but available for Phase 4.

## Color

Tailwind v4 + shadcn default neutral palette (CSS variables in `globals.css`). The 60/30/10 split:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (#ffffff light / #0a0a0a dark) | Page background, body |
| Secondary (30%) | `--card` / `--muted` (#f5f5f5 light / #171717 dark) | Card surfaces, skeleton blocks, input field background |
| Accent (10%) | `--primary` (neutral-900 #171717 light / neutral-50 #fafafa dark) | Submit button background, focus ring on input |
| Destructive | `--destructive` (red-600 #dc2626) | Inline invalid-URL error text only |

Accent reserved for: **Submit button (`<Button>` default variant) and focused input ring only.** No accent color on card borders, headings, links, badges, or backgrounds. The accent is the primary CTA's visual identity — nothing else competes for it.

### Status-to-card color mapping (FETCH-04, FETCH-05, error states)

Phase 3 must visually distinguish status types. Mapping uses shadcn `<Alert>` variants and matches semantic intent:

| Status | Visual treatment | Icon (lucide-react) | shadcn variant |
|--------|------------------|---------------------|----------------|
| `'ok'` (with prType) | neutral card, no banner color | `Shield` (security), `Wrench` (refactor), `ArrowRightLeft` (api-change), `Bug` (bug-fix) | `<Card>` default — no Alert |
| `'size-exceeded'` | warning yellow banner | `AlertTriangle` | `<Alert>` with custom yellow styles: `border-yellow-500/50 bg-yellow-50 dark:bg-yellow-950/30 text-yellow-900 dark:text-yellow-200` |
| `'private-repo'` | info blue banner | `Lock` | `<Alert>` with custom blue styles: `border-blue-500/50 bg-blue-50 dark:bg-blue-950/30 text-blue-900 dark:text-blue-200` |
| `'not-found'` | info blue banner (same as private-repo) | `SearchX` | `<Alert>` with same blue treatment as private-repo |
| `'invalid-url'` | inline destructive text under input — no card | `AlertCircle` (inline) | inline `<p className="text-sm text-destructive">` next to input |

**Rationale for blue/yellow exception to the 60/30/10 rule:** Status color on `<Alert>` banners is informational signaling, not brand accent. It is reserved exclusively for the four error/warning surfaces above and must not appear elsewhere (cards, buttons, headings). The accent role (`--primary`) remains unchanged.

### Dark mode

`globals.css` already uses `prefers-color-scheme: dark`. shadcn-init must keep this scheme (CSS variables flip on `.dark` class OR media query). Phase 3 plan should configure shadcn to use the existing media-query approach to avoid regressing the current dark mode.

## Interaction Contract

| Interaction | Behavior | Timing budget |
|-------------|----------|---------------|
| User pastes URL + presses Enter or clicks `Analyze PR` | Form submits via client `fetch` (or React 19 `useTransition` action) to `/api/analyze-pr` | — |
| Submit fires | Button label switches to `Analyzing…` and disables; result panel renders skeleton | **Skeleton MUST be visible within 100ms of submit (FETCH-03)** — non-negotiable |
| Pipeline resolves OK | Skeleton replaced by `<{PRType}CardPlaceholder />` matching `prType` | 3–8s typical |
| Pipeline resolves with status error | Skeleton replaced by status-specific card (private-repo / not-found / size-exceeded) | 3–8s |
| URL fails client-side regex pre-check | Inline error displays under input; submit blocked; no fetch fires | <50ms |
| Re-submit while analyzing | Button is disabled — no second request can be queued | — |

**Skeleton shape (FETCH-03):**
A `<Card>` containing:
1. One `<Skeleton className="h-6 w-1/3" />` (title placeholder, ~24px tall)
2. Three `<Skeleton className="h-4 w-full" />` rows with `space-y-2` (body lines, ~16px tall each)
3. One `<Skeleton className="h-4 w-2/3" />` (final partial line)

This shape mimics the eventual card layout (heading + 3-4 content rows) so the layout doesn't shift when content lands.

**Focus management:**
- On submit, focus stays on the submit button (do not move focus into result panel — disorienting)
- On error result, do not auto-focus the error message (visually obvious; programmatic focus would steal it from screen-reader users mid-announcement)

## Accessibility Floor

- All form fields have visible labels OR `aria-label`
- Submit button announces state change via `aria-busy="true"` while pending
- Loading skeleton wrapped in `aria-live="polite"` region with screen-reader-only text (`<span className="sr-only">Analyzing pull request…</span>`)
- Color is never the only signal — every status card has an icon + heading + body text alongside its color treatment
- Focus rings inherit shadcn defaults (`focus-visible:ring-2 ring-ring`); do not strip
- Alert components use shadcn's built-in `role="alert"`
